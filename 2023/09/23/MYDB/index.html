MYDB

MYDB

整体结构

MYDB是C/S架构，分为前端和后端，前后端通过socket交互。前端就是读取用户输入的命令行，发送到后端，输出返回结果，等待下一次输入。后端通过解析SQL，如果合法，就尝试执行并返回结果。不合法就抛出异常。

MYDB后端分为五个模块：

  1. 事务管理（Transaction Manager TM）
  2. 数据管理（Data Manager DM）
  3. 版本管理（Version Manager VM）
  4. 索引管理（Index Manager IM）
  5. 表管理（Table Manager TBM）

每个模块职责：

  1. TM通过XID文件维护事务的状态，提供结构给其他模块来进行调用查询事务的状态
  2. DM直接管理数据库DB文件和日志文件。分页管理DB文件，并进行缓存。管理日志文件，保证发生错误的时候可以根据日志进行恢复。抽象DB文件为DatItem供上层模块使用，并进行缓存。
  3. VM基于两段锁实现了调度序列的可串行化，并实现了MVCC以消除读写阻塞。同时实现了读已提交和可重复读两种隔离级别。
  4. IM实现了基于B+树的索引，BTW（范围查找），目前where只支持已索引字段
  5. TBM实现了对字段和表管理，同时，解析SQL语句，并根据语句操作表。

TM 事务管理

XID文件

在XID文件中每个事务占1个字节，每个事务有一个XID，是这个事务的唯一标识。从1开始自增，不可重复。特殊规定XID 为 0 是一个超级事务，当一些事务没有申请事务的时候，可以将XID设置为0。XID为0的事务永远是已提交状态。

TM维护了一个XID格式的文件，记录各个事务的状态。这里有三种状态：

  1. active：正在进行，尚未结束
  2. committed：已提交
  3. aborted：回滚

xid文件格式：

xid文件中头部由一个8字节的数字记录这个文件中事务的个数。所以每个事务就存储在（xid-1）+ 8 的位置。因为超级事务不需要记录，所以 xid-1 才是事务存储的准确位置。

事务的状态表示：

分别使用0、1、2表示未提交、已提交和回滚。

文件读写

对于操作文件读写。使用NIO的随机读写文件RandomAcessFile和文件管道FileChannel进行。

校验XID文件合法性

通过文件头的8字节事务数量反推文件理论长度，和实际文件长度对比。

过程：

  1. 通过TM接口创建xid文件，返回一个TM实现类，包含了随机访问文件的引用和文件通道的引用。
  2. 校验合法性的时候获取文件长度，是否小于8，若小于则抛出异常。
  3. 读取文件头8个字节到缓冲区，转为字节数组，转换为long类型，赋值到一个成员变量，xid的计数器
  4. 通过一个获取当前xid的偏移量的方法获取偏移量
  5. 比较偏移量和实际文件长度，判断是否合法
  6. 如果不合法，直接退出系统，强制停机，无法恢复的错误只能直接停机

DM 数据管理

数据管理是MYDB中最底层的模块。

DM直接管理数据库DB文件和日志文件。

  1. 分页管理DB文件，并进行缓存。
  2. 管理日志文件，保证发生错误的时候可以根据日志进行恢复。

缓存

分页管理和数据项都涉及了缓存。

LRU缓存和引用计数策略？

LRU（Least Recently Used）缓存是一种常见的缓存替换策略，它用于管理缓存中的数据项。

why not LRU?

由于分页管理和数据项（DataItem）管理都涉及缓存，这里设计一个更通用的缓存框架。

看到这里，估计你们也开始犯嘀咕了，为啥使用引用计数策略，而不使用 “极为先进的” LRU 策略呢？

这里首先从缓存的接口设计说起，如果使用 LRU 缓存，那么只需要设计一个 get (key) 接口即可，释放缓存可以在缓存满了之后自动完成。设想这样一个场景：某个时刻缓存满了，缓存驱逐了一个资源，这时上层模块想要将某个资源强制刷回数据源，这个资源恰好是刚刚被驱逐的资源。那么上层模块就发现，这个数据在缓存里消失了，这时候就陷入了一种尴尬的境地：是否有必要做回源操作？

  • 不回源。由于没法确定缓存被驱逐的时间，更没法确定被驱逐之后数据项是否被修改，这样是极其不安全的
  • 回源。如果数据项被驱逐时的数据和现在又是相同的，那就是一次无效回源
  • 放回缓存里，等下次被驱逐时回源。看起来解决了问题，但是此时缓存已经满了，这意味着你还需要驱逐一个资源才能放进去。这有可能会导致缓存抖动问题

问题：LRU策略中，驱逐资源不可控，上层模块无法感知，引用计数策略解决了这个问题。只有上层模块主动释放引用，缓存在确保了没有在使用这个资源了，才会去驱逐资源。

增加一个释放资源引用的方法，上层模块不使用某个资源时，释放对资源的引用。当引用归零时，缓存就会驱逐这个资源。

同时在缓存满了之后，引用计数无法自动释放缓存，此处直接报错（内存溢出，同JVM相似，OOM）

引用计数缓存框架

通过一个抽象类实现缓存，具体实现由继承类去实现

提供两个抽象方法，当资源不在缓存中的获取行为和当资源被驱逐时的写回行为。

抽象类中维护了3个map：

  1. 实际缓存的数据 private HashMap<Long, T> cache;
  2. 资源的引用个数 private HashMap<Long, Integer> references;
  3. 正在被获取的资源 private HashMap<Long, Boolean> getting;

同时成员变量记录了缓存的最大缓存数量和缓存中的元素的个数。

提供获取缓存方法、强行释放一个缓存方法和关闭缓存写回所有资源方法

获取缓存：

  1. 进入无限循环，多线程下需要等待其他线程释放资源
  2. 获取锁
  3. 检查资源是否正在被其他线程获取，如果有释放锁，等待一段时间，然后重新获取资源。确保资源不会被多个线程同时获取
  4. 如果资源已经在缓存中，直接返回资源，增加一个该资源的引用
  5. 如果资源不在缓存中，尝试获取资源，先判断缓存是否已满，如果满了抛出异常，否则标记该资源正在获取中，增加缓存中元素个数，释放锁，跳出循环。
  6. 跳出循环之后，尝试从底层数据源中获取，调用当资源不在缓存中的获取行为的方法。获取过程中若抛出异常，则减少缓存中的资源数量，然后移除正在获取资源的标记，重新抛出异常。否则获取资源成功，再次获取锁，移除对资源的获取标记，放入缓存中，设置引用计数为1，释放锁，返回资源。

强行释放缓存：

  1. 获取锁
  2. 获取该资源的引用减一后的引用计数
  3. 判断是否等于0，如果等于，回源，调用当资源被驱逐时的写回行为方法，移除引用，移除缓存，缓存计数减一。
  4. 如果不等于0，不能释放资源，将引用计数更新回去
  5. 释放锁

关闭缓存，写回所有资源：

  1. 获取锁
  2. 获取缓存中的所有资源uid集合
  3. 遍历资源uid，调用当资源被驱逐时的写回行为方法，删除相关引用，缓存
  4. 释放锁

日志

每次操作底层数据事，都会记录一条日志，方便数据库发生崩溃时恢复，保持数据一致性。

日志读写

日志的二进制文件格式：

[XChecksum][Log1][Log2][Log3]...[LogN][BadTail]

  • XChecksum：4字节整数，对后续所有日志计算的校验和
  • Log1-n：日志常规数据
  • Bad Tail：数据库崩溃时，未写完的日志，不一定存在

每条日志格式：

[Size][Checksum][Data]

  • Size：4字节整数，标识了Data段的字节数，也就是Data的大小
  • Checksum：该条日志的校验和
  • Data：数据

单条日志的校验和，通过一个指定的种子实现：

日志类设计为迭代器模式，一条一条读取日志记录：

  1. 当前日志文件读取到的位置偏移量+数据起始位置偏移量，如果大于文件大小则不能读取下一天，返回空。
  2. 读取从当前读取偏移量的位置读取开始读取，4个字节
  3. 读取到的字节数组转换为整型数组，比较从当前位置偏移量到 size+OF_DATA是否会超出文件大小，超出则返回空。
  4. 否则读取日志数据部分为字节数组
  5. 通过字节数组计算出日志校验和，同时通过日志文件结构中记录的日志大小与其比较，不相同则返回空。
  6. 将当前位置偏移量标志移动到下一条日志的开头。

日志文件中有一个BadTail部分，当打开日志文件时，首先需要校验日志文件的完整性，然后移除损坏的部分：

  1. 首先设置当前偏移量为4
  2. 通过无限循环读取日志文件为一个字节数组，同时校验每条日志的校验和
  3. 然后校验总的日志校验和
  4. 截断除去日志数据意外的其他数据
  5. 设置文件指针的偏移量为4
  6. 再次设置当前偏移量为4，为了回到起始位置

日志恢复策略

DM模块为上层模块提供两种操作，插入新数据，更新现有数据。至于删除操作，使用了逻辑删除，并未提供。

首先日志策略重要的就是：

再进行插入和更新时，需要保证先将插入和删除日志写入日志文件，才进行插入和删除操作

对于两种操作，DM文件记录如下：

  • (Ti,I,A,x) : 表示事务Ti在A位置插入了一条数据x
  • (Ti,U,A,oldx,newx) : 表示事务Ti将A位置的数据，从oldx更新为newx

单线程

在单线程中，事务的日志永远不会相交，所以可以对所有的事务的日志进行重做，判断当前事务状态，如果是已提交或者回滚事务，则进行重做，否撤销。

重做：

  1. 正序扫描所有事务日志
  2. 如果时插入操作(Ti,I,A,x)，重新将x插入A位置
  3. 如果是更新操作(Ti,U,A,oldx,newx) ，就将A位置的值设置为newx

撤销：

  1. 倒序扫描所有日志
  2. 如果时插入操作(Ti,I,A,x)，就将A位置的数据删除
  3. 如果是更新操作(Ti,U,A,oldx,newx) ，就将A位置的值设置为oldx

多线程

多线程有两种情况

第一种：

在系统崩溃时，T2 仍然是活跃状态。那么当数据库重新启动，执行恢复例程时，会撤销 T2，它对数据库的影响会被消除。但是由于 T1 读取了 T2 更新的值，既然 T2 被撤销，那么 T1 也应当被撤销。这种情况，就是级联回滚。但是，T1 已经 commit 了，所有 commit 的事务的影响，应当被持久化。这里就造成了矛盾。所以这里需要保证：

规定1：正在进行的事务，不会读取其他任何未提交的事务产生的数据。

第二种情况：

在系统崩溃时，T1 仍然是活跃状态。那么当数据库重新启动，执行恢复例程时，会对 T1 进行撤销，对 T2 进行重做，但是，无论撤销和重做的先后顺序如何，x 最后的结果，要么是 0，要么是 2，这都是错误的。

出现这种问题的原因, 归根结底是因为我们的日志太过简单, 仅仅记录了”前相”和”后相”. 并单纯的依靠”前相”undo, 依靠”后相”redo. 这种简单的日志方式和恢复方式, 并不能涵盖住所有数据库操作形成的语义

解决：

MYDB 采用的是限制数据库操作，需要保证：

规定2：正在进行的事务，不会修改其他任何未提交的事务修改或产生的数据。

在 MYDB 中，由于 VM 的存在，传递到 DM 层，真正执行的操作序列，都可以保证规定 1 和规定 2。有了这两条规定，并发情况下日志的恢复也就很简单了：

  1. 重做所有崩溃时已完成（committed 或 aborted）的事务
  2. 撤销所有崩溃时未完成（active）的事务

在恢复后，数据库就会恢复到所有已完成事务结束，所有未完成事务尚未开始的状态。

页面索引

缓存了每一页的空闲空间。用于在上层模块进行插入操作时，能够快速找到一个合适空间的页面，而无需从磁盘或者缓存中检查每一个页面的信息。

MYDB 用一个比较粗略的算法实现了页面索引，将一页的空间划分成了 40 个区间。

在启动时，就会遍历所有的页面信息，获取页面的空闲空间，安排到这 40 个区间中。insert 在请求一个页时，会首先将所需的空间向上取整，映射到某一个区间，随后取出这个区间的任何一页，都可以满足需求。

首先根据所需空间大小选择合适空间：

  1. 通过所需空间大小计算出所需区间大小所在的编号
  2. 如果区间编号小于最大区间编号，自行+1，确保所需空间大小超出最大间隔的情况也能被处理
  3. 循环找出当前间隔中是否有适合的空间，如果当前列表为空，就寻找下一个，如果不为空，则移除当前列表，并返回。
  4. 否则找不到就返回空

被选择的页，会直接从 PageIndex 中移除，这意味着，同一个页面是不允许并发写的。在上层模块使用完这个页面后，需要将其重新插入 PageIndex

VM 版本管理

VM 基于两段锁协议实现了调度序列的可串行化，并实现了 MVCC 以消除读写阻塞。同时实现了两种隔离级别。

2PL与MVCC

冲突和2PL

数据库的冲突，暂时不考虑插入操作，只看更新和读取。

对同一个数据库的冲突，就是有两种情况：

  1. 两个不同事务的更新操作事务
  2. 两个不同事务的更新、读取操作

那冲突和不冲突的意义？交换两个互不冲突的操作的顺序，不会对最终的结果造成影响，如果如果两个操作相互冲突，则是会有影响的。

VM就是实现了调度序列的可串行化，采用两段锁协议实现。譬如，T1已经因为U1（x）锁定了x，那么T2对x的读取和写的操作都会被阻塞，T2就必须等待T1释放锁。

问题：事务之间相互阻塞，甚至可能导致死锁。

解决：实现MVCC，降低阻塞概率。

MVCC

DM层向VM层提供了数据项（DataItem）的概念。VM通过管理数据项，向上层提供了记录（Entry）的概念。

上层模块通过操作VM的最小单位（记录）。VM则为每一个记录维护了多个版本（Version）。每当上层模块对某个记录修改时，VM就会为这个记录创建一个新的版本。

譬如，T1想要更新记录x的值，于是T1通过获取锁，操作x，也就是创建了一个x的版本，假如是x3，此时T1还没有释放锁，T2想要读取x的值，这时就不会阻塞，数据库返回一个较老版本给T2，例如是x2。这样T2就会再T1的先执行，保证了可串行化，同时也降低了阻塞概率，注意是降低，不是消除。

数据恢复

在VM层同样需要保证数据的可恢复，所以VM层传递到DM层的操作序列需要满足两个规则：

  1. 正在进行的事务，不会读取其他任何未提交事务产生的数据
  2. 正在进行的事务，不会修改其他任何未提交的事务修改和产生的数据

因为实现了2PL和MVCC，所以这两个条件很容易就满足了。

版本可见性

如果一个记录的最新版本被加锁，当另一个事务想要修改或读取这条记录时，MYDB 就会返回一个较旧的版本的数据。这时就可以认为，最新的被加锁的版本，对于另一个事务来说，是不可见的。

IM 索引管理

IM，即 Index Manager，索引管理器，MYDB提供了基于 B+ 树的聚簇索引。目前只支持基于索引查找数据，不支持全表扫描。

IM 直接基于 DM，而没有基于 VM。索引的数据被直接插入数据库文件中，而不需要经过版本管理。

二叉树索引

二叉树由一个个 Node 组成，每个 Node 都存储在一条 DataItem 中。结构如下：

[LeafFlag][KeyNumber][SiblingUid]
[Son0][Key0][Son1][Key1]...[SonN][KeyN]

  • LeafFlag 标记了该节点是否是个叶子节点；
  • KeyNumber 为该节点中 key 的个数；
  • SiblingUid 是其兄弟节点存储在 DM 中的 UID。

后续是穿插的子节点（SonN）和 KeyN。最后的一个 KeyN 始终为 MAX_VALUE，以此方便查找。

Node 类持有了其 B+ 树结构的引用，DataItem 的引用和 SubArray 的引用，用于方便快速修改数据和释放数据。

节点中是否为叶子节点占1字节，键值对数量占2字节，兄弟节点指针占8个字节，总的节点头部总大小为11个字节。其中平衡因子为32，通常用于决定节点是否需要分裂，节点大小等于节点头部大小+(2*8)每个键和值的大小+(键值对的最大数+2可能有一个额外的一个键和一个子节点)，节点结构如下：

可能的错误和恢复

B+ 树在操作过程中，可能出现两种错误，分别是节点内部错误和节点间关系错误。

当节点内部错误发生时，即当 Ti 在对节点的数据进行更改时，MYDB 发生了崩溃。由于 IM 依赖于 DM，在数据库重启后，Ti 会被撤销（undo），对节点的错误影响会被消除。

如果出现了节点间错误，那么一定是下面这种情况：某次对 u 节点的插入操作创建了新节点 v, 此时 sibling(u)=v，但是 v 却并没有被插入到父节点中。

这时，如果要对节点进行插入或者搜索操作，如果失败，就会继续迭代它的兄弟节点，最终还是可以找到 v 节点。唯一的缺点仅仅是，无法直接通过父节点找到 v 了，只能间接地通过 u 获取到 v。
