在Agent系统里，向量模型有啥用？

前几篇我们分别讲解了：

  • [Agent 理论基础（一）模型与能力]
  • [Agent 理论基础（二）从一次回答，到会推进任务的 Agent]
  • [Agent 理论基础（三）Agent如何推进任务]
  • [Agent 理论基础（四）AI如何真正做事]

我们已经解决了一个关键问题：AI 如何请求系统“做事”。

通过工具调用和 MCP，Agent 已经可以：

  • 查询数据库
  • 调用接口
  • 发送邮件
  • 操作真实系统

但只要你真正开始写 Agent，很快就会撞上另一个问题。

一、AI 怎么知道“该怎么查”

我们继续沿用前面那个贯穿始终的任务：

  • 查询最近一周的用户评论
  • 对评论内容进行分析
  • 输出结论或发送邮件

为了完成这个任务，我们可能已经给 Agent 提供了一个工具：

于是问题来了：

AI 要写什么 SQL？

要写一条正确的 SQL，至少需要知道：

  • 有哪些表
  • 表之间的关系
  • 哪张表存评论
  • 时间字段叫什么
  • 用户字段在哪

但这些信息，对模型来说并不是“理所当然存在的”。

二、一个必须直面的事实：模型不知道你的系统

无论模型多强，它都不可能天然知道：

  • 你的数据库结构
  • 你的业务表名
  • 你的字段命名规范
  • 你的内部流程
  • 你系统里那些“只存在于你这里”的规则

这些内容有一个共同点，它们不是通用知识，而是系统知识。而模型的训练数据主要来自通用世界。所以，如果在没有任何额外信息的情况下，让 AI 直接生成 SQL，最常见的结果只有两种：

  • 写出一条“看起来合理、但执行失败”的 SQL
  • 反复试错，浪费大量 token 和时间

这不是能力问题，而是信息缺失问题。

三、最直觉、但很快会失败的方案

一个非常自然的想法是：“那我把所有数据库结构都塞进提示词里不就行了？”

于是你可能会尝试：把所有表结构，所有字段说明，所有业务备注，一股脑丢给模型。

这个方案在小系统里，可能还能勉强工作，但只要系统稍微复杂一点，就会立刻遇到现实限制：

  • 数据库结构本身很长
  • 提示词有长度限制
  • 成本不可控
  • 每次任务只需要其中极小一部分信息

更关键的是：即便你把所有信息都给了 AI，它也并不需要全部。

真正的问题其实是，在“所有系统知识”中，哪一小部分，和当前这个任务是相关的？

四、向量模型，终于开始登场

这正是向量模型发挥作用的地方。

我们在第一章中已经提到，向量模型只做一件事：衡量内容之间在语义层面上是否相关。

它并不理解业务逻辑，也不生成答案，但它非常擅长解决一个关键问题：“这些内容，和当前问题像不像？”

五、把系统知识，变成“可检索的文本”

一旦换一个视角，事情就会变得非常清晰。我们可以把系统中那些：

  • 数据库表结构说明
  • 字段含义
  • 业务规则
  • 内部流程文档

整理成一段段文本，然后使用向量模型，把这些文本转成向量，存入向量数据库，在需要时，按语义进行检索。

这样一来，当 Agent 接到一个任务，比如：“查询最近一周的用户评论并统计问题类型”，系统并不会立刻让 AI 去写 SQL，而是先在“系统知识库”中，找出和“评论、时间、用户”最相关的那几段说明。

六、RAG：不是让模型记住，而是让模型“查到”

当系统把检索到的相关信息作为上下文注入到当前任务中，再让模型基于这些信息去生成 SQL、做分析、做决策，这整套流程，就是我们常说的：

RAG（Retrieval-Augmented Generation，检索增强生成）

RAG 的关键价值，并不在“增强生成”这四个字上，而在于“检索”。

它解决的是一个非常现实的问题：模型不需要记住整个系统，它只需要在需要的时候，看见该看的那一页。

七、RAG 更多细节

在很多介绍中，RAG 常常被描述成：“给模型加一个长期记忆”，“让 AI 记住你的私有数据”，这种说法并不完全错，但很容易产生误解。

一个更准确的理解是：知识库不是模型的记忆，而是系统为 AI 准备的一本“随用随查的说明书”。

这些知识不属于模型本身，不需要被“学会”，也不需要被永久记住，它们只是在当前任务需要时，被系统检索出来临时提供给模型参考。

当用 RAG 的方式组织系统知识，会立刻获得几个非常实在的收益：

  1. **上下文更短：**只给当前任务相关的信息
  2. **准确率更高：**模型基于真实结构做决策
  3. **成本可控：**不再反复试错
  4. **知识可维护：**改文档即可，不用改模型
  5. **系统边界清晰：**模型不“拥有”知识，系统拥有

前面我们主要用“数据库结构”来举例，但在真实系统中，RAG 解决的问题远不止这些。任何具有以下特征的信息，都非常适合通过 RAG 提供给 AI：

  • 业务规则
  • 产品设计约定
  • 内部流程
  • 系统之间的依赖关系
  • 各种不通用、只存在于你系统中的知识

它们的共同点是不会出现在通用模型的训练数据中，却是 AI 在当前场景下“必须知道”的信息。

八、把 Agent 的能力链条连起来

到这里，我们终于可以把前几章的内容，完整地连成一条链：

  • 模型：负责理解与生成
  • Agent：负责目标与过程推进
  • 工具 / MCP：让 AI 能影响真实系统
  • 向量模型 + RAG：让 AI 在需要时，知道该知道的事

这并不是在“让 AI 无所不知”，而是在用系统的方式，弥补模型不知道的那一部分内容。
