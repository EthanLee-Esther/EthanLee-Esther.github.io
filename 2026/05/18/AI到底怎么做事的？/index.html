AI到底怎么做事的？

**Agent 的价值，不在于“多聪明”，而在于“能不能把任务推进下去”。**但只要任务涉及真实世界，就会立刻遇到一个无法回避的问题：AI 到底是怎么“做事”的？

本章我们讲一讲：

  • 一、AI 再聪明，也只是“想”
  • 二、模型和系统的天然分工
  • 三、最原始、也最直接的方式：工具调用
  • 四、当工具越来越多，问题就不只是“调不调”
  • 五、MCP：把“工具”变成可管理的能力
  • 六、AI 得知道“该用哪个工具”

一、AI 再聪明，也只是“想”

我们先回到一个非常基本的事实。无论模型看起来多智能，它都无法：

  • 真的查询数据库
  • 真的发送邮件
  • 真的调用内部系统
  • 真的修改文件

原因很简单：**这些行为，本质上都是程序在执行代码，模型所做的一切，始终停留在“生成内容”**这一层：

  • 它可以生成一条 SQL
  • 可以生成一段接口参数
  • 可以生成“我应该发送一封邮件”的判断

但如果没有真实系统配合，这些都只是“想法”。

二、模型和系统的天然分工

一旦接受了这个事实，整个 Agent 系统的分工就会变得非常清晰：

  • 模型负责判断与决策
  • 系统负责真实执行

换句话说，**AI 决定“该不该做、做什么”，程序决定“怎么做、能不能做”，**这条分工线，是所有 Agent 系统稳定运行的基础。

三、最原始、也最直接的方式：工具调用

为了让 AI 能影响真实系统，我们需要给它一条“合法出口”，最简单的方式就是：提前写好一组函数，把它们作为“工具”提供给 AI。

例如：

  • 查询数据库 → queryDatabase(sql)
  • 发送邮件 → sendEmail(to, content)

这些函数的内部实现：

  • 怎么连数据库
  • 怎么做鉴权
  • 怎么处理异常

AI 完全不需要知道，也不应该知道。

AI 只需要知道三件事：

  1. 有哪些工具
  2. 每个工具是干什么的
  3. 需要什么参数

然后再给 AI 一条非常关键的规则：如果在完成任务过程中，你认为需要使用某个工具，请告诉我，要调用哪个工具，以及对应的参数是什么。

当 AI 按照约定，返回工具名称和参数结构后，系统就可以：

  1. 解析 AI 的返回
  2. 调用真实的函数或接口
  3. 获取执行结果
  4. 再把结果原样交还给 AI

AI 再根据这些真实结果，继续推进任务，这种协作方式，就是我们通常所说的：Tool Calling 或 Function Calling，它的核心思想并不复杂，但非常重要：AI 不执行代码，它“请求”系统执行代码。

四、当工具越来越多，问题就不只是“调不调”

一旦系统中工具开始增长，你很快会遇到一组结构性问题：

  • 工具越来越多，怎么统一描述？
  • 每个工具需要哪些参数？
  • 新增一个工具，要不要改一堆提示词？
  • 工具只能是我项目里的函数吗？
  • 能不能用别人已经写好的工具？

如果这些问题都靠“临时约定”解决，**工具本身就会变成新的复杂源头。**这时候，问题已经不再是“怎么写函数”，而是：怎么管理工具这个概念本身。

在早期的工具调用设计中，工具往往等价于项目里的一个函数或某个类的方法，但随着 Agent 系统复杂度提升，一个非常重要的变化开始出现：工具，不一定非得存在于你的代码里。

从系统角度看：

  • 查询数据库，是一种能力
  • 发邮件，是一种能力
  • 调用第三方接口，也是一种能力

至于这项能力是本地函数、远程服务、还是别人写好的现成系统，对 Agent 来说，都不重要。重要的是：能力如何被描述，如何被发现，以及如何被安全地调用。

五、MCP：把“工具”变成可管理的能力

正是在这样的背景下，MCP 这个概念出现了。

你可以把 MCP 理解为一套约定与运行方式，用来解决一个核心问题：**如何把工具，从“项目内部的函数”， **变成“可被 AI 统一使用的外部能力”。

在 MCP 的模式下，工具可以是独立运行的服务，可以由其他团队提供，甚至可以直接复用别人已经实现好的工具只要遵循同一套约定，Agent 就可以“看见”并使用它们。

这里发生了一个非常关键的转变：Agent 不再依赖“我写了什么函数”，而是依赖“系统对外暴露了什么能力”。

如果把这一章的内容串起来，会发现一条非常清晰的演化路径：

  1. 一开始，只是想让 AI 能“调用函数”
  2. 接着，希望工具能被更好地描述和约束
  3. 再往后，希望工具可以被复用、共享、独立运行
  4. 最终，把工具抽象成“能力”，并通过统一协议暴露

Tool Calling 解决的是“能不能调用”，而 MCP 关注的是“能力如何被组织和管理”。

六、AI 得知道“该用哪个工具”

到这里，我们其实已经解决了“AI 如何请求系统做事”的问题。但还有一个更根本的问题，尚未解决：AI 怎么知道，在当前任务下，该用哪个工具？

比如：

  • 要查询数据库，写哪张表？
  • 要分析数据，用哪种结构？
  • 哪些信息和当前任务是相关的？

如果 AI 对系统一无所知，再好的工具也用不好。

而这正是下一章要讨论的核心内容：向量模型、向量数据库，以及 RAG。

在下一章中，我们将看到：工具解决的是“能不能做”，RAG 解决的是“知道不知道”。

两者结合，Agent 才真正具备“做事”的能力。
