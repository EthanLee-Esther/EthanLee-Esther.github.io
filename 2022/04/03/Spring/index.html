Spring 学习笔记

Spring

基础

spring是什么？特性？有哪些模块？

轻量级、非入侵式的控制反转和面向切面的框架

特性：

  1. IOC和DI支持
  2. 声明式事务支持
  3. 快捷测试
  4. 快速集成
  5. 复杂API模板封装

模块：

  • Spring core：核心基础，提供IOC和依赖注入DI特性
  • Spring Context：上下文容器，BeanFactory的加强子接口
  • Spring Web：提供web开发支持
  • Spring Mvc
  • Spring Dao
  • Spring ORM：支持流行的ORM框架整合
  • Spng AOP

IOC

什么是IOC？什么是DI？

IOC（控制反转）：对象的生命周期和依赖关系交给容器管理，我们需要什么，容器就给我什么。

DI（依赖注入）：容器在把对象实例化时将它依赖的对象注入给它

简单说一下IOC的实现机制？

IOC就是让容器去管理对象，自动化生成对象，我们需要什么就拿什么。

  1. 首先创建一个bean的配置文件，配置文件中指定bean名字和bean所对应的类。
  2. 定义bean的定义类，对应配置文件中bean的实体
  3. 定义资源加载器，加载配置文件中类和bean的信息
  4. 定义对象注册器，通过对象名字获取对象，创建一个map集合作为缓存，默认为单例，若存在则直接从map中获取，不存在则通过名字和对象放入map中
  5. 定义对象工厂类，初始化时创建了对象的注册器，完成了资源的加载。通过名字获取对象，首先从缓存中拿，若没有，则创建对象，传入类，通过反射获取类的实例和名字，创建出来放入缓存中。

说说BeanFactory和ApplicantContext?

我的理解是BeanFactory是心脏，ApplicantContext就是身躯。

首先ApplicantContext就是BeanFactory派生来的，在BeanFactory基础上提供了更多的功能，像bean的实例化/串联、统一的资源文件访问方式、同时加载多个配置文件、支持国际化这些。

至于他们的区别就是从四个方面来说：

  • 加载方式：
    BeanFactory是延迟加载注入，只有第一次使用的时候才会实例化，导致有时候不能发现一些配置问题，第一次使用的时候才抛错
    ApplicantContext是在容器启动时就一次创建所有对象，预载入所有单例对象。
  • 创建方式：
    BeanFactory是编程式创建
    ApplicantContext除了编程式创建，还可以声明式创建 ContextLoader
  • 注册方式：
    两者都支持BeanPostProcessor、BeanFactoPostProcessor的使用，但是BeanFactory支持手动注册，ApplicantContext是自动注册
  • 依赖关系
    BeanFactory是spring最底层的接口，提供了对象定义、配置文件读取、加载、对象实例化、生命周期控制、维护依赖关系等一系列功能。
    ApplicantContext除了BeanFactory拥有的功能以外，还提供额外功能：
    - 统一资源文件访问方式
    - 同时加载多个配置文件
    - 继承Messageresource，支持国际化
    - 在监听器中注册bean的事件

Spring 容器启动阶段会干什么？

启动时分为两个阶段：容器启动阶段和Bean实例化阶段。

其中容器启动阶段是加载配置文件中的信息到bean定义中。

实例化阶段就是实例化对象，装配依赖，生命周期回调。

容器启动开始，首先会通过一些工具类像BeanDefinitionReader解析加载的configuration metadata，将分析后的信息加载到对象定义类中，通过对象定义类注册到对象定义注册器里就完成了容器的启动。

SpringBean的生命周期？

Bean的生命周期大致分为四个阶段：实例化、属性赋值、初始化、销毁。

实例化：实例化一个bean对象
属性赋值：设置bean的相关属性和依赖
初始化：初始化通过调用BeanNameAware设置对象名，接着设置对象工厂，调用前置初始化处理方法，初始化bean，自定义init方法，后置初始化处理方法，使用bean。
销毁：DisposableBean销毁方法，自定义销毁方法。

Bean定义和依赖定义有哪些方式？

有三种方式：

  1. 直接编码
  2. 配置文件
  3. 注解
  • 直接编码：一般用不到，其他方式最终都是通过直接编码的方式实现
  • 配置文件：通过xml、properties等配置文件配置相应的依赖关系，读取配置文件，完成相关依赖注入
  • 注解：用的最多的一种方式，扫描注解，完成依赖注入

Spring 有哪些自动装配的方式？

什么是自动装配？

自动装配就是Spring IOC容器知道所有的bean信息，通过反射机制可以获取实现类的构造方法、属性等信息。通过这些信息就可以对容器中的bean自动装配，无需显式进行依赖配置。

自动装配方式：

  • byName：根据名字装配
  • byType：根据类型自动装配
  • constructor：与类型装配相似，但是是针对构造方法而言
  • autodetect：根据Bean的自省机制来决定是byType还是constructor，如果Bean提供了默认的构造函数，则通过bytype，否则通过 constructor

Spring 中的Bean的作用域有哪些？

五种作用域：

  • singleton：仅存在一个实例，Bean以单例形式存在，默认作用域
  • prototype：每次从容器中调用bean时，都回获取一个新的实例
  • request：每一次http请求都会产生一个新的Bean，只在这一次请求中有效
  • session：一个session中共用一个bean，不同的session不同的Bean
  • globalSession：同一个全局session使用同一个Bean

Spring 中的单例Bean会存在线程安全问题吗？

首先Spring中的单例Bean不是线程安全的。

因为是单例Bean，所有线程共享。如果说单例Bean是无状态的（不做除了除了查询意外的任何操作），那么这个单例就是线程安全的。如果这单例是有状态的会对Bean中的成员变量进行读写操作，那么就可能存在线程安全问题。

如何解决这些单例bean的线程安全问题？

  1. 将Bean定义为多例（不推荐）
  2. 将Bean中的成员变量保存到Threadloal中

说说循环依赖？

什么是循环依赖？

循环依赖就是自己依赖自己，或者a中依赖b的，b中依赖a的。

Spring 中支持解决哪些循环依赖？

AB 循环依赖：

  • ab均采用构造器注入，不支持
  • ab均采用setter方法注入，支持
  • ab均采用属性自动注入，支持
  • a中注入的b为setter注入，b中注入的a为构造器注入，支持
  • b中注入的a为setter注入，a中注入的b为构造器注入，不支持（原因是spring创建bean时默认按照自然排序创建，a先于b创建）

如何解决循环依赖的问题？

Spring提供了三级缓存解决

  • 一级缓存：单例池，专门保存实例化、属性赋值（注入）、初始化完成的bean实例
  • 二级缓存：早起曝光对象，用于保存实例化完成的bean实例
  • 三级缓存：早起曝光对象工厂，用于保存bean创建工厂，以便于后面扩展有机会创建代理对象

AB循环依赖过程：

  1. 创建A实例：实例化A时，将A实例化对象工厂放入三级缓存，虽然不完整，暴露出来让大家知道。
  2. A注入属性时，发现依赖B，B还没有被创建，去实例化B，实例化B的时候发现依赖A，从一级缓存找到三级缓存，三级缓存发现了对象工厂A，虽然不完善，但是存在，把A放入二级缓存，删除三级缓存的A，B实例化完成。
  3. 接着A完成属性赋值，在一级缓存中拿到B，A也完成了初始化，将二级缓存中的A删除，A放入一级缓存。
  4. 最后AB都完成实例化和初始化存在一级缓存。

为什么要三级缓存？二级不行吗？

不行，主要是为了生成代理对象，如果没有代理对象，使用二级缓存也是没有问题的，但是如果存在代理，三级缓存没有问题，二级缓存就不行了。

三级缓存中存放的是生成的具体对象的匿名内部类、获取Object的时候，他就可以生成代理对象，也可以返回普通对象。使用三级缓存是为了每次从中拿到的都是同一个对象。

如果只有二级缓存的话，往二级缓存中放一个普通bean对象，bean初始化过程中使用BeanPostProcessor生成代理对象之后，覆盖掉二级缓存中的普通Bean对象，就可能导致对象不一致。

@Autowired 实现原理？

实现@Autowired的功能，主要也是通过后置处理器AutowiredAnnotationBeanPostProcessor。

创建Bean的过程中，调用doCreateBean（）方法，在此方法中调用populateBean（）方法进行属性填充，完成自动装配等工作。
populateBean（）方法中调用了两次后置处理器，第一次是为了判断是否需要属性填充，如果不需要，直接return，如果需要继续往下执行，调用postProcessPropertyValue（）方法解析Autowired注解，实现自动装配。

查找@Autowired，现在容器中找，结果为一个时，直接装配，不止一个时，根据名称查找，不存在会抛出异常。解决：required = false

AOP

说一说什么是AOP?

面向切面变成，就是把业务逻辑中一些相同的代码抽取到一个单独模块中，让业务逻辑更加清爽。

一、定义

AOP（Aspect-Oriented Programming，面向切面编程） 是一种编程范式，旨在将那些横跨多个模块的通用逻辑（称为”横切关注点”）从业务代码中分离出来，以实现代码的模块化和解耦。

二、核心概念

术语 说明
切面 横切关注点的模块化封装，如日志、事务、权限等
连接点 程序执行中的某个点，如方法调用、异常抛出
切入点 定义”在哪些连接点上”应用切面的表达式
通知/增强 切面在连接点上执行的动作，包括：Before、After、Around、AfterReturning、AfterThrowing
织入 将切面应用到目标对象并创建代理的过程

三、为什么需要 AOP？

❌ 没有 AOP 时（代码臃肿、耦合严重）

🤯 每个业务方法都充斥着日志、权限、事务等非核心代码，散落各处，难以维护。

✅ 使用 AOP 后（职责清晰、优雅解耦）

🎯 通过注解或配置，将横切关注点统一管理，业务代码变得纯净。

四、五种通知类型

通知类型 执行时机 典型场景
Before 目标方法执行前 参数校验、权限检查
AfterReturning 目标方法正常返回后 记录返回结果
AfterThrowing 目标方法抛出异常后 异常日志、告警
After 目标方法执行后（无论是否异常） 资源释放
Around 包裹目标方法（最强大） 性能统计、事务管理

五、AOP 的实现方式

方式 说明 代表框架
编译时织入 在编译阶段修改字节码 AspectJ
类加载时织入 类加载时修改字节码 AspectJ + Agent
运行时动态代理 运行时生成代理对象 Spring AOP（JDK动态代理 / CGLIB）

🔔 Spring AOP 默认使用 JDK 动态代理（目标实现了接口时）或 CGLIB（目标未实现接口时）。

六、经典应用场景

  • 📝 日志记录：统一记录方法入参、出参、耗时
  • 🔐 权限校验：统一鉴权拦截
  • 💾 事务管理：Spring @Transactional 的底层实现
  • ⚡ 性能监控：方法耗时统计
  • 🔄 缓存管理：方法结果缓存
  • 🛡️ 异常处理：统一异常捕获与转换
