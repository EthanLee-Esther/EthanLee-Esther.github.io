登陆注册

设计方案

注册登录除了常见的用户名+密码的登录方式之外，现在也有流行的手机号+验证，第三方授权登录；我们最终选择微信公众号登录方式

对于个人公众号，很多权限没有；因此这个登录的具体实现，有两种实现策略

  • 点击登录，登录页显示二维码 + 输入框 -> 用户关注公众号，输入 “login” 获取登录验证码 -> 在登录界面输入验证码实现登录

  • 点击登录，登录页显示二维码 + 验证码 -> 用户关注公众号，将登录页面上的验证码输入到微信公众号 -> 自动登录

其中第一种策略，类似于手机号/验证码的登录方式，主要是根据系统返回的验证码来主动登录

优点：

代码实现简单，逻辑清晰

缺点：

操作流程复杂，用户需要输入两次

对于第二种策略，如果是企业公众号，是可以省略输入验证码这一步骤的，借助动态二维码来直接实现扫码登录；对于我们这种个人公众号，则需要多来一步，通过输入验证码来将微信公众号的用户与需要登录的用户绑定起来

工作内容

通过 AOP + TraceID 记录接口访问日志，实现任务的追踪、监控和诊断。

该项目中使用AOP的共有两处，一处是用于方法执行耗时统计，另一处是动态切换数据源。

方法耗时统计

参照SkyWalking中的SkyWalkingTraceldGenerator类生成 TraceID。

生成的TraceID包含了三部分：

  1. 第一部分是应用实例id，它是在类加载时生成的uuid，对于每个进程，它是唯一的。
  2. 第二部分是当前线程的id
  3. 第三部分是一个由时间戳和线程序列号组成的数字，时间戳是毫秒级的，而线程序列号是一个在0到9999之间的数。

项目中自己实现的是包含了四部分：

  • IP地址（8位）：获取当前机器IP地址，并将其转换为十六进制格式
  • 时间戳（13位）：使用Java8的Instant类获取当前的毫秒级时间戳。
  • 进程号（5位）：使用Java的ManagementFactory类获取JVM进程的PID，并保证总长度为5位。。
  • 自增序列号（4位）：一个在1000到9999之间循环自增的数。

通过slf4j的MDC上下文诊断映射，存储traceid的信息。通过拦截器在每一个请求中添加全链路traceId

通过自定义注解MdcDot，自定义注解中有值为业务编码，默认为空，定义切面和切点实现记录每个方法的执行效率。这里采用环绕通知，首先获取当前系统时间，然后检查是否有MdcDot注解并获取业务编码，解析业务编码对应方法参数中的值返回。放入MDC中，只有MDC中有值才执行成功，计算结束时间，清空MDC。

总结就是，MDC存储全链路traceId用于追踪，监控，还要存储带有自定义注解方法的业务编码，用于计算执行时间，写入日志，清空业务编码。

动态数据源切换

主要借助AbstractRoutingDataSource 来实现动态数据源

  • defaultTargetDataSource ：标识默认使用的数据源
  • Map<Object,Object> targetDataSources：动态数据源可以根据key来切换使用的数据源
  • determineCurrentLookupKey：返回当前数据源使用的标识

首先是多数据源配置，通过spring的dynamic.primary来指定默认启用的数据源。

定义一个数据源配置加载类，借助spring的ConfigurationProperties来实现读取加载默认数据源和多数据源配置。

当配置了多数据源时，启用数据源初始化，通过配置文件中加载的数据源配置，装配到目标数据源targetDataSources中，过程中会根据不同的类型选择数据源，其中包括了一个默认数据源。最后返回一个自定义的路由数据源，重写了determineCurrentLookupKey，可以通过数据源持有上下文获取注解的标识，从而选择数据源。

通过自定义注解，有两个值，一个是默认主库，一个是启用的数据源，如果有启用的数据源则优先使用。

定义切面和切点，使用环绕通知，可用于类和方法上。通过获取类或者方法上的所有自定义注解，将注解中的标识放入数据源持有上下文中。接着就能执行方法，执行方法是就会通过上下文中的数据源标识选择目标数据源中的对应的数据源。执行成功之后清空上下文中的数据源标识。

遇到的问题
  1. 此处数据源持有的上下文实现是直接使用线程上下文实现的，类别是string类型，当一个方法是有数据源选择注解时，此时方法内部开了子线程做异步处理，那么上下文存储的数据源会失效
  2. 一个方法上指定了数据源，它又调用其他也指定了的数据源的方法，当前的方案会存在数据源的选择冲突
  3. druid数据源不兼容

问题一和二：

上下文赋值只能赋值一次，为了解决，想到的是使用栈的数据结构–后进先出，实现嵌套的数据源切换方案

定义一个数据源node节点来存储数据源，它除了存储当前选中的数据源外，还存储上一次选择的数据源。将上下文中的string类型替换为数据源节点类型。使用InheritableThreadLocal继承线程上下文，实现主子线程数据共享。

每当有一个新的数据源选中时，新创建一个节点，并重新加入上下文中，之前的节点作为当前新节点的上一个节点。

清空上下文时也要注意，按照栈的方式，如果没有选中的数据源直接返回，如果存在上一个节点，则将上一个节点重新写入上下文，若不存在，清空上下文。

微信公众号实现自动登录

用户在前端点击登录时，会展示公众号二维码和带输入的验证码，并会和前端构建一个半长连接，同时保存验证码和半长连接之间的映射关系。

当用户向公众号发送验证码时，微信公众号会将用户发送信息转发给服务器，然后通过验证码来识别请求登录的用户身份。

由于验证码保存了和半长连接的映射关系，所以可以找到该半长连接，实现用户的自动登录跳转。

这里微信公众平台会返回一个用于唯一标识的uuid，在用户表中存储一个第三方用户id。唯一用户识别。

获取登录验证码时，实现了两个缓存，一个是验证码和半长连接之间的映射关系，另一个是设备id和验证码之间的映射关系。并设置过期时间为5分钟，当获取一个不存在设备id的验证码，通过循环不断生成验证码，直到是一个唯一的验证码。保证同一个设备唯一性，验证码的唯一性和有效性。

添加设备id是保证同一个设备用于不断刷新页面时，不断向验证码和半长连接映射添加kv对，但是保证验证码依然是五分钟时效，且验证码不会变（刷新除外）。

登录成功之后生成session并保存，在filter过滤器中将用户信息存入上下文，用于用户身份识别。

将消息发送到RabbitMQ，实现消息异步解耦，提升系统效率和服务稳定性

消息队列主要有两种模式，“点对点模式” 和 “发布订阅模式”。

点对点模式

一个具体的消息只能由一个消费者消费。且一个消息被处理的时候，该消息会被锁住或者被移除并且其他消费者无法处理该消息。如果处理失败一般会放回消息队列，其他消费者可以继续处理。

发布订阅模式

单个消息能被多个订阅者并发的获取和处理。

分为临时订阅（消费者存在才存在）和持久订阅（一直存在，除非主动删除）

原理

AMQP协议

具有现代特征的二进制协议，应用层协议的一个开放标准，提供统一消息服务的应用层标准高级消息队列协议。

工作原理

AMQP协议由三部分组成生产者、消费者和服务端。

  1. 生产者连接到server，建立一个连接，开启一个信道
  2. 生产者声明交换机和队列，设置相关属性，并通过路由键将交换机和队列进行绑定
  3. 消费者也需要建立连接，开启信道等操作，便于接受信息
  4. 生产者发送消息，发送到服务端中的虚拟机
  5. 虚拟机中的交换器根据路由键选择路由规则，发送到不同的消息队列中。
  6. 订阅了消息队列的消费者就可以获取到消息，进行消费。
    ![[Pasted image 20230920210312.png]]

四种常用交换器：direct（点对点）、topic（通配符匹配）、fanout（发布订阅）、headers（根据头部信息进行匹配）

项目中使用了RabbitMQ的点对点交换机，在mq配置类中实现ApplicationRunner接口，重写run方法，实现mq的初始化。考虑到每次的连接创建都要new一个并发量起来的话会持续消耗内存，机器肯定扛不住的。所以创建了mq的连接池，使用基于链表实现的阻塞线程安全队列Blockingqueue实现。

所以发送消息过程就是从连接池拿到连接，创建信道，声明交换机，发送消息，将连接归还连接池
消费消息就是：从连接池拿到连接，创建信道，确认消息队列，绑定队列到交换机，接收并消费消息，将连接归还连接池。

对于文章的点赞、收藏、评论这些会走mq的消息机制，同时持久化到db中。

通过Redis实现计数统计和用户活跃度排行，并通过Redis + MySQL提高系统并发

用户活跃度

这里使用redis的一个zset数据结构实现，zset是redis的一个有序的数据结构，同时set能保证数据的唯一性，利用zset自带一个权重功能来实现排序，系统中的话就是score，就是每当用户在执行的一些操作，比如说点赞、评论或者是发布文章这些时，我们都会对用户进行一个加分的操作。通过redis的incr指令去进行加分操作，我们在进行用户更新活跃度的时候会进行一个幂等判定，防止重复添加，若幂等了直接返回，否则，执行更新，并做好幂等保存。这里的幂等策略是每个用户维护一个活跃更新操作历史记录表，直接将用户的历史日志，保存在redis的hash数据结构中，每天一个记录。

  • key：activity_rank_{user_id}_{年月日}_
  • field：活跃度更新key_
  • value：添加的活跃度

但是这里存在一个并发问题，幂等无法百分比百生效，可能存在重复添加或者扣减的情况。多次操作redis存在事务问题。

并发中可能存在两个用户同时加分或者减分，就会造成资源抢占的问题，一开始是通过幂等，但是我们的系统在后续的过程中，如果要实现一个分布式这样一个情况的话，一开始设计了一个解决这种并发的方案。就是通过redis分布式锁。

首先一开始使用redis的setnx和setex，来保证这样一个原子性的操作。我在更新用户活跃度的时候要么同时成功，要么同时失败。比如果我们对一个数据进行查询的时候通过缓存查询，缓存中没有就去数据库中查询，访问缓存到数据库这样一个过程中就会存在并发的问题。数据库的效率本来就没有缓存高，资源是比较紧张的，如果如果这时大量数据涌入就会出现数据库宕机的情况，所以我们要对这个过程进行加锁。

对于redis实现分布式锁中的问题就是一个过期时间的问题，就是业务执行完之后，但是我们的锁还被持有着，虽然有过期时间和redis的淘汰策略，但是还是不太好，redis对于我们这个系统还是很重要的。

所以考虑到了一种解决方案是通过在finally模块，执行完业务之后我们去释放锁，但是释放锁有会存在一个问题，就是释放锁的时候会释放别人的锁，也就是a线程执行过程中，时间过期，锁被自动释放，b线程就会获取到该锁，执行业务逻辑。但是此时线程A需要释放锁，所以释放的是B的锁。所以再次解决，设置一个value值，来防止是防止释放其他锁。因为每个用户的userid对应一个key，当key进来时设置一个value值，在执行finally块的时候通过先比较value值再进行释放锁就能保证不会释放其他锁。

但是问题又来了，过期时间如何设置，过期时间过短，可能业务还没有完成，锁就被释放了。过期时间太长的话，枷锁成功之后没有执行到释放锁，节点宕机了，那么锁未过期这段时间其他线程不能够获取。

解决通过实现一个守护线程，每隔一段时间去看看业务是否执行完毕，如果没有执行完毕就延长过期时间，也就是锁续期。

针对以上情况选择了redisson实现分布式锁，redission实现分布式锁解决了锁过期问题，还有释放他人的锁，它还是可重入锁。它的内部机制是过期时间是30秒，然后会有一个定时任务每10秒查询一次锁是否被释放，如果没有释放就延长30秒。
如果没有获取到锁，将while循环获取继续加锁。主要就是这么一个设计过程。也写了相应的代码。
更新完用户活跃度之后，我们通过redis的zrevrange进行倒序排序，因为我们更新完之后都是从小到大的，倒序排序去前30名作为一个数据的展示。

redisson还存在一个问题，就是高一致性，redisson是无法解决高一致性的。可以通过红锁或者zk锁，但是会丢失高可用性，对用户体验不好。

计数统计

如果要查询用户统计信息，我们的用户信息是通过redis的hash数据结构来实现，key为用户id，值为用户的关注数、已发布文章数、文章阅读数这些。通过redis就能查询所有的用户统计信息，然后通过定时任务每天更新统计信息，确保信息的一致性。

redis + mysql 的使用，存在缓存一致性的问题。开始设计了两种方案：

  • 实时一致性方案：是先写MySQL，再删除redis，这种虽然也会存在两者不一致的问题，但是条件有点苛刻，所以是满足实时性条件下，能尽量满足一致性的最优解。
  • 最终一致性方案： 先写mysql，通过binlog，异步更新redis，可以通过binlog，结合消息队列异步更新redis，是最终一致性最优解。

考虑到项目的实时性要求高，选择实时一致性方案处理。所有涉及到db更新操作都需要按照这种方式处理。如果redis删除失败，MySQL的更新需要添加事务回滚，避免查询读取到脏数据。

对于用户的统计计数和用户的活跃度数据是通过rabbitmq来获取的。

使用DFA算法实现铭感词过滤，防止一些不文明的词汇出现

DFA算法数据结构是一个DFA树，准确来说应该是森林，每个节点代表一个字符，从根节点到叶子节点就是一个完整的敏感词。

项目中选用了一个 sensitive-word 作为库，使用基于mybatis拦截器的自定义数据库脱敏实现方案。

通过自定义注解，放在需要脱敏的数据库实体对象的成员上，实现查询拦截器，返回数据库内容到数据库实体对象上时，若判断有对应的注解，则赋值敏感词替换后的内容。

具体实现层，添加了一个缓存，存储添加了注解的数据库实体对象的成员，减少每次对实体成员的判定。

mybatis拦截器原理

通过拦截注解intercepts指定拦截的方法，我们拦截的时结果处理器中的处理结果方法。拦截到结果集之后通过查询缓存中是否有对应元数据，如果没有则回调构造带有自定义注解的元数据，然后放入缓存中。然后通过遍历元数据列表替换敏感词，如果是字符串则直接替换，如果是集合类型和非基本类型的对象，递归替换。
